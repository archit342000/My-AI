# RAG Directives

## Overview

The RAG (Retrieval-Augmented Generation) system provides semantic and keyword search capabilities across global memory and chat-specific files. It uses a hybrid search strategy to maximize retrieval accuracy across different document types.

## Architecture

The system uses **ChromaDB** as the vector store and implements a singleton `RAGManager` to coordinate embedding and storage.

### Data Model: Dual-Collection Strategy

For every logical store (e.g., `file_store`), the system maintains two physical collections:

1.  **`{store_name}_vector`**: Stores custom embeddings generated by the embedding model (e.g., `embeddinggemma`, 768-dim). Used for semantic retrieval.
2.  **`{store_name}_bm25`**: Stores default internal ChromaDB embeddings (384-dim). Used for keyword/lexical matching.

### Hybrid Search & Scoring

Retrieval combines results from both collections using **Reciprocal Rank Fusion (RRF)**:

- **Vector Search**: Calculates L2 distance (`hnsw:space`: `l2`).
- **BM25 Search**: Provides keyword overlap scores.
- **Fusion Logic**:
  $$ Score = \frac{1}{k + rank_{vector}} + \frac{1}{k + rank_{bm25}} $$
  *(Where $k$ is a constant, usually 60).*

## RAG Optimization (Grid Search)

To maintain high retrieval quality (Recall@K and MRR), the system includes a parameter optimization pipeline.

### Core Metrics

| Metric | Description |
|--------|-------------|
| **Recall@K** | % of relevant documents found in the top K results. |
| **MRR** | Mean Reciprocal Rank - measures how high the first relevant result appears. |
| **NDCG** | Normalized Discounted Cumulative Gain - accounts for rank position of all relevant items. |

### Optimization Parameters

All optimization parameters are centralized in `backend/config.py`:

- `RAG_MIN_SEMANTIC_SCORE`: Threshold to filter out irrelevant vector matches.
- `RAG_FETCH_MULTIPLIER`: Number of candidates to fetch before fusion.
- `RAG_DEDUP_THRESHOLD`: Cosine similarity threshold for removing near-duplicate chunks.
- `RAG_DECAY_RATE`: Time-based decay for older documents.

## Rules for Agents

### Rule 1: Use `RAGManager` Singleton

**ENFORCED:** Direct instantiation of `RAGManager` is **strictly forbidden** and will raise a `RuntimeError`.

Always use the provider pattern:

```python
from backend.providers import RAGProvider

# Initialize once at startup
rag_manager = RAGProvider.get_manager(
    persist_path=config.CHROMA_PATH,
    api_url=config.EMBEDDING_URL,
    embedding_model=get_embedding_model(),
    api_key=config.EMBEDDING_API_KEY
)

# All RAG consumers receive the manager as a parameter
research_rag = ResearchRAG(rag_manager=rag_manager)
file_rag = FileRAG(rag_manager=rag_manager)
file_manager = FileManager(rag_manager=rag_manager)
```

**Never do this:**
```python
# WRONG - this raises RuntimeError:
rag_manager = RAGManager(persist_path=config.CHROMA_PATH)
```

**Provider behavior:**
- Config is set on the **first** call to `get_manager()`
- Subsequent calls return the same instance and ignore any config parameters
- Use `RAGProvider.reset()` to reset state (useful for testing)

### Rule 2: Multi-Type Support
Chunks are tagged with `file_type`. Agents should use these tags in `where` filters (e.g., `file_type: "code"`) to improve precision.

### Rule 3: Parameter Fetching
Never hardcode RAG thresholds. Always fetch from `config.py` to allow for environment-specific tuning.

### Rule 4: Hard Limit Guarantee
**Absolute Compliance**: All chunking strategies MUST strictly enforce the `max_tokens` limit. If a monolithic unit (e.g., a massive row, a giant URL, or a block of data with no spaces) exceeds the limit after all natural splitting attempts (paragraphs, sentences, words), it MUST be subdivided using a **recursive character-based split**. Mathematical compliance with the embedding model context window is prioritized over structural continuity.

## Operational Commands

- **Reset Store**: `rag_manager.reset_store(store_name)` deletes all collections for a given namespace.
- **Migration**: When changing embedding models, the internal `collection_metadata` tracks the current model name to detect and prompt for re-indexing.

### RAG Parameter Optimization (Grid Search)

The optimization pipeline runs automated evaluations against massive parameter sets. It strictly relies on the output of `tests/fetch_real_test_data.py`.

*   **Isolated Storage:** The test overrides `detect_file_type` using `file_type_override` to ensure documents are ingested using perfectly known ground-truths, preventing classifier error from masquerading as retrieval error.
*   **Latin Hypercube Sampling (LHS):** Rather than evaluating 6,500+ combinations, the grid searches 512 stratified samples to achieve statistical guarantees while radically minimizing API load.

**How to Run**:
```bash
docker compose -f docker/docker-compose.testing.yml up --build rag_grid_search
```

**Evaluation Script**: `tests/test_rag_parameters.py`
This script spawns parallel worker processes, runs evaluations based on Reciprocal Rank Fusion, and dumps comprehensive Recall/MRR metrics into the `reports/` folder in both JSON and Markdown formats.
